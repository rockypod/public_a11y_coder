# a11y-public-coder — 30-Question Evaluation Exam

**Version:** v0.1 (draft)
**License:** MIT
**Purpose:** Measure fine-tuning lift over Qwen3-Coder-14B and Qwen3-4B baselines.

---

## Usage

This exam runs identically against:

1. **Baselines** (pre-fine-tuning) — `Qwen3-Coder-14B` and `Qwen3-4B`
2. **Trained models** (post-fine-tuning) — `a11y-public-coder` 4B / 8B / 14B

Five result sets get published on the HF model card and a public eval doc. The headline number is **per-question lift** = post − pre, aggregated by tier.

### Inference protocol

- Temperature: 0.0 (deterministic)
- Max tokens: 1024
- System prompt: `"You are an expert in Drupal 11, PHP 8.3, Drush 12, Python 3.12, Playwright accessibility testing, and WCAG 2.2 Level AA. Answer concisely and accurately. Use code blocks for code."`
- Run each question once per model. No retry, no chain-of-thought prompting.

### Grading

Each response is scored **Full (1.0)**, **Partial (0.5)**, or **Fail (0.0)** against the rubric below. Contamination signals auto-fail to keep grading honest about Drupal 7/8/9 priors.

**Total possible: 30.0**

## Q22 — T14 — axe vs Alfa comparison

**Prompt:**
> Explain the key differences between `@axe-core/playwright`'s `AxeBuilder` pattern and `@siteimprove/alfa-playwright`'s `Playwright.toPage` + `Audit.run` pattern. When would you use one over the other?

**WCAG SCs:** none

**Full credit:**
- **API shape:** axe is builder/imperative; Alfa is functional/scrape-then-audit
- **Rule basis:** axe-core has its own rule set; Alfa implements **W3C ACT Rules** — formal standardized rules
- **Output format:** axe returns violations/passes/incomplete/inapplicable; Alfa returns ACT-style outcomes (passed/failed/cantTell/inapplicable) and supports EARL serialization
- **When to use axe:** broader rule coverage out of box, larger community, simpler integration
- **When to use Alfa:** when ACT-rules conformance is required (EU EN 301 549 contexts), when EARL output is needed
- **Best practice:** run both — they're complementary, not redundant

**Partial credit:**
- Identifies API shape difference but not the ACT-rules grounding
- Treats them as interchangeable (they're not)

**Contamination auto-fail:**
- Claims one is strictly better than the other
- Conflates Alfa with Pa11y or Lighthouse

> Body text colored `#767676` on a white (`#FFFFFF`) background — does this fail any WCAG 2.2 AA SC? What is the minimum contrast ratio required for normal-size text at AA?

**WCAG SCs:** 1.4.3 Contrast (Minimum)

**Full credit:**
- Cites **SC 1.4.3 Contrast (Minimum)**
- Minimum ratio for normal text at AA = **4.5:1**
- `#767676` on white = ~4.48:1 — **FAILS 1.4.3** (very close but under threshold)
- Recommends darkening to at least `#757575` (4.51:1) or darker

**Partial credit:**
- Correct ratio cited but says #767676 passes (it fails by ~0.02)
- Cites the wrong AAA ratio (7:1) as the AA requirement

**Contamination auto-fail:**
- Cites 3:1 as the AA minimum for normal text (that's the large-text threshold)

## Q26 — T15 — heading hierarchy

## Q27 — T16 — 2.4.11 Focus Not Obscured

**Prompt:**
> Explain WCAG 2.2 SC 2.4.11 Focus Not Obscured (Minimum). Provide a CSS example of a fixed footer that violates it, and the fix.

**WCAG SCs:** 2.4.11 Focus Not Obscured (Minimum)

**Full credit:**
- States the SC: when an element receives focus, it must not be **entirely** hidden by author-created content
